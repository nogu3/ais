# CLAUDE.md

`ais` — Panasonic AiSEG2 専用 CLI。AiSEG2 の Web UI を HTTP で叩き、電力データの読み取りと、AiSEG2 経由でしか操作できない機器（リンクプラス照明等）の制御を行う。AI ネイティブなスマートホーム CLI 群の一員。

> 名前: **`ais`**（AiSEG2）。**パブリックの独立リポジトリ**。
> 対象は **AiSEG2 のみ**。AiSEG2 は 2026年9月生産終了・以後ファーム更新なしのため、**HTML/制御エンドポイントは凍結する**前提で、リバース対象のファーム版に固定する。AiSEG3 は別物（対象外。将来やるなら別 CLI か別モード）。
> パブリックのため、自宅の実機 IP・回路名・機器 ID・ユーザー名/パスワードを **テスト/サンプル/コミットに含めない**。フィクスチャは必ずサニタイズし、IP は `192.0.2.0/24`（RFC 5737）等のダミーを使う。

---

## プロジェクトの目的と立ち位置

enl（ECHONET Lite）や mat（Matter）が機器プロトコルを直接喋るのに対し、AiSEG2 は**製品**であり、その操作面は Web UI しかない。リンクプラス照明のように **AiSEG2 経由でしか自動化できない機器**が存在し（ECHONET Lite で届かず、公式 API も IFTTT も非対応）、`ais` はその唯一の経路を担う。

**`ais` がやること:**
- AiSEG2 の Web ページをフェッチ（HTTP Digest 認証）し、HTML をパースして電力データを構造化 JSON で出す。
- AiSEG2 の機器コントロール（制御 AJAX/POST）を叩いて、対応機器を操作する。

**`ais` がやらないこと:**
- 他プロトコル/他製品。SwitchBot・Matter・ECHONET Lite は別 CLI（横断は casa）。
- スケジューリング・ポーリング・常駐。1 実行 = 1 スナップショット or 1 操作。
- 人間向けの整形。見せ方はダッシュボード側の責務。
- LLM 呼び出し。

| 対象 | CLI 名 | 状態 |
|---|---|---|
| ECHONET Lite | `enl` | 開発中 |
| SwitchBot | `sbl`（仮） | 未着手 |
| Matter | `mat`（候補） | 設計中 |
| AiSEG2 | `ais` | 本リポジトリ |

横断オーケストレーションは casa が担う。`ais` は casa から `PATH` 上のバイナリとして叩かれることを主な利用形態とする。

## この対象（AiSEG2）特有の前提

1. **壊れやすい契約が 2 つある。** codec ではなく、(a) 読みページの **HTML→値のパース**、(b) 制御の **AJAX/POST エンドポイント＋ペイロード**。両方ともファーム依存。**両方を独立モジュールに隔離する**（= enl の codec 隔離に相当）。
2. **その契約は凍結する。** AiSEG2 は生産終了・更新停止のため、固定したファーム版の構造は今後変わらない。リバース対象のファーム版を README に明記する。
3. **制御可能機器の真実は AiSEG2 が持つ。** 何を on/off できるかはハードコードせず、`ais devices`（機器コントロール一覧の取得）を真実とする。

---

## 絶対に守る設計原則（変更時は要相談）

1. **ステートレス / one-shot**
   デーモン化しない。ポーリングループを持たない（過度なアクセスは機器に負荷をかける）。コマンドは「接続 → リクエスト → タイムアウト付きで応答 → 終了」で完結。定期実行は呼ぶ側（cron / n8n / ダッシュボード）の責務。
2. **stdout は純粋な構造化 JSON のみ**
   進捗・ログ・人間装飾を混ぜない。`jq` と LLM Function Calling がそのまま食えること。1 コマンド = 1 JSON。
3. **診断は stderr に構造化ログ（`tracing`）**
   ネットワークエラー・タイムアウト・認証失敗・パース失敗・制御リジェクトを切り分けられる粒度で stderr へ。
4. **クレデンシャル・設定を内部に持たない**
   host / user / pass は引数か環境変数（`AISEG_HOST` / `AISEG_USER` / `AISEG_PASS`）で受ける。設定ファイルを所有しない。名前→機器のマッピングは casa 側。パスワードはシェル履歴に残さないため env を第一とする。
5. **フェッチ層と解釈層を分離**
   HTTP/Digest を扱う層と、HTML パース・制御ペイロード生成を扱う層を分ける。ファームで構造が変わったら**解釈層だけ**を直せる状態を保つ。
6. **パース失敗を握りつぶさない**
   セレクタが当たらない＝ファームがずれた可能性。専用の exit code（後述）で呼ぶ側に通知する。

## 技術スタック

| 領域 | 選択 | 備考 |
|---|---|---|
| 言語 | Rust | enl と同じ |
| CLI parser | `clap`（derive） | |
| HTTP | `ureq`（blocking） | tokio を引かない。enl の std 優先と整合 |
| 認証 | 自前 Digest（MD5, qop=auth） | 50 行程度。依存を増やさない |
| HTML パース | `scraper`（CSS セレクタ） | 解釈層に隔離 |
| JSON 出力 | `serde` + `serde_json` | |
| ログ | `tracing` + `tracing-subscriber` | stderr へ |

依存は小さく保つ。コア業務は「ページを取る／POST する → 構造化する」だけなので過剰設計しない。

## アーキテクチャ

```
casa / cron / n8n / dashboard
        │  Command::new("ais")
        ▼
       ais
   ┌────┴─────┐
   │ フェッチ層 │  HTTP + Digest（ureq）
   └────┬─────┘
        ▼
   ┌──────────┐
   │  解釈層   │  HTML パース / 制御ペイロード生成（ファーム依存はここに閉じ込める）
   └────┬─────┘
        ▼
   stdout = JSON ／ stderr = 構造化ログ
        ▲
        │
   AiSEG2（Web UI, Digest 認証）
```

## 規約

### stdout
- 成功時、結果を JSON で stdout に出す。読みも制御結果も同じ規律。
- AiSEG2 の生 HTML をそのまま流さない。`ais` のスキーマに正規化して出す。

### stderr
- エラーは構造化ログ：`{"error": {"kind": "...", "detail": "..."}}`。
- `kind` 例: `auth_failed` / `network` / `timeout` / `http_status` / `parse_failed` / `device_not_found` / `control_rejected`。

### exit code
| Code | 意味 |
|---|---|
| 0 | 成功 |
| 2 | CLI 引数エラー（clap） |
| 3 | ネットワーク / タイムアウト |
| 4 | 認証失敗（401） |
| 5 | AiSEG2 が想定外 HTTP ステータスを返した |
| 6 | パース失敗（セレクタ不一致＝**ファームがずれた可能性**） |
| 7 | 制御リジェクト / 結果未確認 |
| 11 | 指定機器が機器コントロール一覧に見つからない |

`6` を独立させることで、ダッシュボード側が「`ais` のパースが壊れた」を即検知できる。

## スコープ

### 読み
- 分電盤：主幹 + 分岐回路ごとの電力
- 太陽光発電
- 買電
- 電気使用量（総消費）

### 制御
- 照明（リンクプラス）on / off
- エアイー / 天井埋込形空気清浄機 on / off（**機器コントロール一覧に出る場合のみ**）

### 対象外（重要）
- 電動シャッター / カーテン → **enl 経由**で扱う。`ais` に入れない。
- 電気錠 → **SESAME（別経路）。read-only 運用のため `ais` の制御対象にしない。**
- 給湯系（エコキュート / エネファーム / ガス給湯器）・EV 充電・床暖房など、物理的副作用が重い機器 → v1 では扱わない。
- 宅外操作・シーンの自動起動など AiSEG2 アプリ固有の高機能。

---

## サブコマンド（実装順）

1. `ais power` — 瞬時値（太陽光発電 / 買電 / 電気使用量）を JSON で。
2. `ais circuits` — 分電盤（主幹 + 分岐回路）を JSON 配列で。
3. `ais fetch <page>` — 任意ページを取得して解釈層に通す読み用エスケープハッチ。
4. `ais devices` — 機器コントロール一覧（制御可能機器とその状態）を JSON で。
5. `ais on <device>` / `ais off <device>` — 照明・エアイー等の制御。
6. （任意）`ais scene <name>` — リンクプラスのシーン制御。

## ロードマップ

### Phase 0 — 読み（フェッチ層 + パース層）
**ゴール**: 分電盤と電力が JSON で取れる。「わかりやすく見れる」素材がここで揃う。
- フェッチ層（ureq + Digest）。
- `ais power`、`ais circuits`。
- **保存した実 HTML フィクスチャ（サニタイズ済み）に対する統合テスト**。CI に実機は不要。
**完了条件**: `cargo test` がフィクスチャでパースを検証。`kind` 値が文書化されている。

### Phase 1 — 制御
**ゴール**: 機器コントロールを叩いて照明等を操作できる。
- 制御 POST を実機セッションから devtools でキャプチャ → 制御層を実装 → POST フィクスチャでテスト。
- `ais devices`、`ais on` / `ais off`。
- 制御の成功/リジェクトを exit code `7` で区別。
**完了条件**: ダミー応答に対する制御層テスト。実機 E2E 手順を README に記載（CI には載せない）。

### Phase 2 — 拡張
- `ais scene`、積算系（日/月）など、必要が出たら追加。

## やらないこと

- ポーリング / 監視ループ（呼ぶ側の責務）。
- デーモン化・常駐・内部スケジューラ。
- 設定ファイル・クレデンシャルの内蔵（env / 引数のみ。名前マッピングは casa）。
- シャッター/カーテン制御（enl 経由）。
- 電気錠（SESAME 別経路）・給湯系・EV 等の高副作用機器。
- AiSEG3 対応（別 CLI / 別モード）。
- 人間向け整形・LLM 呼び出し。
- 実機 IP / 回路名 / 機器 ID / パスワードのコミット。
